设计模式学习笔记

Posted on 2019-08-06

设计模式六大原则

单一职责原则

一个类应该仅有一个引起它变化的原因

开闭原则

  • 扩展开放，修改关闭
  • 在程序需要拓展的时候，不修改原有的代码，而是增加新的类
  • 提高程序扩展性，易于维护和升级
  • 基于接口和抽象类

里氏代换原则

  • 任何基类可以出现的地方，子类一定可以出现
  • 只有当派生类可以完全替换掉基类且系统功能不受影响时，基类才能真正被复用
  • 对开闭原则的补充

依赖倒转原则

  • 针对接口编程，依赖于抽象而不依赖于具体
  • 开闭原则的基础

接口隔离原则

  • 使用多个隔离的接口优于使用单个接口
  • 降低类之间的耦合度

迪米特法则（最少知道原则）

  • 一个实体类应当尽量少的与其他实体类之间发生相互作用
  • 系统功能模块相对独立

合成复用原则

  • 多使用合成，少用继承

模版模式

  • 别名：模版方法模式

  • 意图：

    定义一个操作中的算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。

  • 使用时机：

    1. 有一些通用的方法，多个子类共有，且逻辑相同
    2. 重要的，复杂的方法可以考虑作为模版方法

  • 优点：

    1. 封装不变部分，扩展可变部分
    2. 提取公共代码，便于维护
    3. 行为由父类控制，子类实现

  • 缺点：

    不同的方法都需要一个子类进行实现，类的个数增加，系统更加庞大

  • 示例图

    由示例图可以看出来，抽象基类定义了基本方法play并使其无法被覆盖

    同时，定义了其他三个抽象方法（忘记加abstract了），从而对行为进行约束，方法的具体实现则由子类负责

    俺也不清楚这个和实现接口哪个更合适来着

工厂模式

  • 意图：

    定义创建对象的接口，由子类确定具体实例化哪一个工厂类

    将对象的创建过程延迟到子类进行

  • 使用场景：

    • 主要解决接口选择的问题
    • 当明确不同条件下创建不同实例时使用

  • 优点：

    1. 调用者想创建某一个对象时，只需要知道对象的名称
    2. 扩展性高，如果想要增加一个产品，只要扩展一个工厂类就可以了
    3. 屏蔽产品的具体实现，调用者只关心接口

  • 缺点：

    每次增加产品都需要增加工厂类，系统中类的个数成倍增加，一定程度上增加了系统的复杂度，同时也增加了系统具体类的依赖

抽象工厂模式

抽象工厂模式是围绕一个超级工厂创建其他的工厂，该超级工厂为其他工厂的工厂

  • 意图

    提供一个创建一系列相关或相互依赖对象的接口，而无需指定他们具体的类

  • 使用场景

    • 系统的产品有多于一个产品族，系统只消费其中某一族的产品

    • 生成不同操作系统的程序

  • 优点

    当一个产品族中的多个对象被设计成一起工作时，它能保证客户端始终只使用同一个产品族中的对象

  • 缺点

    产品族扩展非常困难，要增加一个系列的某一产品，既要在抽象的creator里边增加代码，又要在具体的视线里边增加代码

单例模式

  • 创建型模式
  • 单一类，负责创建自己的对象，且确保只有耽搁对象被创建

  • 注意：

    • 单例类只有一个实例
    • 单例类必须自己创建此唯一实例
    • 单例类必须对外提供此唯一实例

  • 意图：

    保证一个类只有一个实例且可以全局访问此实例

    主要用于解决一个全局使用的类频繁的创建与销毁的问题，控制实例数目，节省系统资源

  • 关键：

    构造函数私有化

  • 优点

    1. 内存里只有一个实例，减少内存开销
    2. 避免对资源的多重占用（写文件操作）

  • 缺点

    没有接口，不可继承，违反单一职责原则

建造者模式

  • 将构建的基本组件与构建过程相分离
  • 基本组件不改变，组合方式发生变化
  • 使用场景：
    • 需要生成的对象有复杂的内部结构
    • 需要生成的对象内部属性本身相互依赖
  • 注意：建造者模式更关注零件装配的顺序

原型模式

在运行期建立和删除原型，意图用原型实例指定创建对象的种类，并通过拷贝原型创建新的对象

  • 使用场景

    • 资源优化场景
    • 类的初始化需要加载较多资源（数据，网络资源）
    • 性能，安全性要求
    • new初始化对象时需要较多的数据准备或访问权限
    • 单一对象多个修改者，多个调用的问题
    • 一般搭配工厂方法模式，由工厂方法将clone出来的对象提供给调用者

  • 优点：

    提高性能，规避构造函数的约束

  • 缺点：

    1. 必须实现clonneable接口
    2. 配置克隆方法时要全面考虑类的功能，对已有的类较难进行配备，不支持串行化或含有循环调用时较麻烦

适配器模式

  • 意图：

    将一个类的接口转化为客户希望的另外一个接口，使原本不兼容接口兼容

  • 使用场景：

    有动机的修改一个正常运行的系统的接口，目标类必须为抽象类

    1. 系统需要使用现有的类，此类的接口不符合系统要求
    2. 建立一个可重复使用的类，用于使一些没有太大关联的类一起工作
    3. 通过接口转换将一个类插入另一个类系中

  • 优点：

    1. 提高了类的复用
    2. 两个无关类可以一起运行
    3. 增加了类的透明度
    4. 灵活度高

  • 缺点

    1. 适配器的使用使系统零乱不好把控
    2. java单继承机制，至多一个适配者类，目标类必须是抽象类

    确实是在修改已有系统内容时比较合适的设计思路，可以避免对已有代码的修改，通过增加新的中间类完成功能的实现

桥接模式

  • 意图

    分离抽象与实现，使其可以独立的变化

  • 使用场景

    • 系统可能有多个维度分类，可能有多种变化的情况
    1. 系统需要构件的抽象化角色和具体化角色之间增加更多的灵活性，避免两个层次之间建立静态的继承联系，通过桥接模式在抽象层建立一个管理关系
    2. 不希望使用继承或多层次继承导致系统类的个数急剧增加的系统，桥接模式尤为适用
    3. 一个类存在两个独立变化的维度，且这两个维度都需要扩展

  • 解决方式

    把多个角度分离出来，以独立的类进行描述，使其独立变化，减少耦合

  • 优点：

    1. 抽象和现实分离
    2. 优秀的扩展能力
    3. 实现细节对客户透明

  • 缺点：

    增加系统的理解与设计难度，要求开发者针对抽象进行编程

过滤器模式

  • 个人感想：

    1. 定义过滤器接口——criteria，此接口具有一个过滤方法——filter
    2. 根据需要的规则编写对应的实现了criteria接口的类，实现过滤方法
    3. 当需要根据多个不同规则进行过滤的时候，可以通过AndCriteria，OrCriteria类实现criteria接口，此连接类可持有两个不同的criteria接口实现，并在过滤方法中基于与或非的思路以不同的顺序和判断条件调用不同的过滤器类的filter方法

组合模式

  • 个人感想：

    以二叉树的形式，将具有层级关系的一组对象组成成为一个整体，可通过一个单一对象进行访问

    使用的结构为树形结构

    在树之中的每一个对象都可以视为树的节点

装饰器模式

可以理解为python中的装饰器

外观模式

  • 作用：

    提供系统的入口接口，为子系统一组接口提供一个高层次接口

  • 优点：

    1. 减少系统相互依赖
    2. 提高了系统的灵活性
    3. 提高系统安全性

  • 缺点：

    不符合开闭原则，修改麻烦，继承，重写都不方便

  • 使用场景：

    1. 为复杂的模块或子系统提供外界访问的模块
    2. 子系统相对独立
    3. 预防低水平人员带来的风险
